ОПИСАНИЕ РАБОТЫ ПРОЕКТА "FFT_class_test".

Переменные, которые возможно менять до запуска проекта в makefile:
   NUMBER_OF_LENGTHS;
   LOW_EDGE;
   HIGH_EDGE.

Подробности
   Вся работа проекта осуществляется запуском команды "make" в make-утилите под управлением ОС "Windows".
   При запуске "make", в хронологическом порядке, происходит следующее.
 
   1. "FFT_class_test.cpp" автоматически генерирует разные длины последовательностей, заключая их значения
      в vector-контейнер. Автоматически производится расчёт прямого и обратного преобразований Фурье, после
      чего осуществляется запись результатов абсолютной ошибки текущего С++-скрипта в файл "Results.txt".
      В параллель, для сравнения полученных результатов с теорией, производится запись комплексных элементов
      векторов последовательностей в файл "Complex_vectors.txt".
   
   2. Результаты с Complex_vectors.txt подгружаются в python-скрипт, который, на их основе, автоматически
      производит подсчёт прямого и обратного преобразований Фурье и записывает, как продолжение, результаты
      в файл Results.txt.
   
   3. Срабатывает правило clean, которое удаляет объектный и исполняемый файлы, участвовавших в роли
      промежуточных.
   
   Таким образом, при совместной работе "FFT_class_test.cpp" и "FFT_class_test.py" в файле "Results.txt"
   формируются конечные данные, на основе которых можно сделать заключение о корректности работы класса FFT
   и его методов из "FFT_class_test.cpp". При каждом запуске "make" генерируемые данные уникальны и
   обновляются в "Complex_vectors.txt" и "Results.txt". Ход работы "FFT_class_test.cpp" иллюстрируется
   в логах терминала. Проект был выполнен в Microsoft VSCode на базе компилятора MinGW64.

Потенциальные проблемы
   Компьютер оснащён 16 ГБ ОЗУ. Исходя из этого при больших значения HIGH_EDGE, смысл которой отображён
   в комментарии makefile, возникает Stack Overflow и Segmentation fault в файле new_allocator.h.
   Segmentation fault обусловлена переполнением vector-аллокаторов. Stack Overflow решается выделением памяти
   под стек в 2 ГБ. Поэтому, из-за недостатка памяти в качестве предельного значения переменной HIGH_EDGE было
   взято число 300. На Ваших машинах оно может быть больше. При соблюдении этого условия проект работает
   корректно.

Предлагаемое решение
   Нужно доработать алгоритм, переписав его основательно. Но это не даст гарантий того, что потенциальная
   проблема будет решена. Нужно больше оперативной памяти.
